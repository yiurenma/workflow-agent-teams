## 1. Overview

Rewrite `DeployModal.tsx` to replace the three-step operation-API deployment sequence with a single call to the online API. The UI assembles the request payload from props already available in the component (`currentSettings` + `currentWorkflow`) and delegates all deployment logic to the pre-configured workflow on the online API.

---

## 2. Current Behavior

**File:** `workflow-ui/src/components/DeployModal.tsx`

The existing modal:
1. Collects 5 user inputs: Deploy URL, Application Name, Username, Password, Environment
2. Constructs a Basic Auth header from Username + Password
3. Executes 3 sequential `fetch()` calls against the user-supplied Deploy URL:
   - `POST /workflow/entity-setting?applicationName=...`
   - `PATCH /workflow/entity-setting?applicationName=...`
   - `POST /workflow?applicationName=...`
4. Shows a 3-step progress indicator

**Problem:** Requires the user to know the target environment URL and service account credentials. The deploy logic is baked into the UI and duplicates what the online API workflow can do.

---

## 3. New Behavior

**File:** `workflow-ui/src/components/DeployModal.tsx`

### 3.1 Form Fields

| Old (5 fields) | New (1 field) |
|---|---|
| Deploy URL | ~~removed~~ |
| Application Name | **Execution Application Name** (pre-filled, editable) |
| Username | ~~removed~~ |
| Password | ~~removed~~ |
| Environment | ~~removed~~ |

### 3.2 Payload Assembly

```typescript
const confirmationNumber = crypto.randomUUID();
const body = JSON.stringify({
  entitySetting: currentSettings,  // Block A — source app entity settings
  workflow: currentWorkflow,        // Block B — full workflow data
});
```

### 3.3 API Call

```typescript
await onlineApi.postWorkflow({
  applicationName: applicationName.trim(),  // user-entered execution name
  confirmationNumber,                        // auto-generated UUID
  body,
  contentType: "application/json",
});
```

This uses the existing `onlineApi` service (`src/api/services/online.ts`) which:
- Routes to `ONLINE_API_BASE` (`/api/proxy/online` → `https://workflow-online-api-nr3e4.ondigitalocean.app`)
- Adds `X-Request-Correlation-Id` header automatically
- Passes `applicationName` and `confirmationNumber` as query params

### 3.4 Request to Online API

```
POST /api/workflow?applicationName=<executionName>&confirmationNumber=<uuid>
Content-Type: application/json
X-Request-Correlation-Id: <auto-generated-uuid>

{
  "entitySetting": { ...WorkflowEntitySettingRow... },
  "workflow": { "pluginList": [...], "uiMapList": [...] }
}
```

## 4. Component Changes

### 4.1 `workflow-ui/src/components/DeployModal.tsx`

**Changed:**
- Removed `DeployFormData` interface (5-field form state)
- Removed `DeployProgress` / `STEP_LABELS` (3-step progress)
- Removed `isCrossOrigin()` / `buildUrl()` helpers (CORS proxy logic)
- Removed `deployToRemote()` (3 sequential fetch calls)
- Added `applicationName` state (pre-filled from props, replaces form)
- Added `deployStatus: "idle" | "success" | "error"` state
- Added `useEffect` to sync `applicationName` when modal opens
- Calls `onlineApi.postWorkflow()` with `{entitySetting, workflow}` body
- Progress indicator replaced with success/error banner

**Imports added:**
- `import { onlineApi } from "@/api/services/online";`

### 4.2 Other Files

No changes required. The existing `onlineApi.postWorkflow()` service and proxy config handle routing.

## 6. Security Considerations

- No credentials (username/password) are handled in the browser — the online API uses its own auth
- `confirmationNumber` is generated client-side as a UUID — no security concern
- Payload contains application configuration data already visible in the Hub UI
